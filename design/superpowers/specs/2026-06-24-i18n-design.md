# Healerbook 国际化（i18n）P1 设计

> **状态**：设计已与用户确认，待落实施计划。
> **日期**：2026-06-24
> **分支**：feat/i18n

## 1. 背景与目标

Healerbook 当前所有面向用户的文案以中文硬编码在源码中，无任何 i18n 基础设施。本设计为项目搭建**可持续的国际化基础设施**，并完成 **1 个样板文件**的文案抽取，作为后续批量抽取的范式。

- **目标语言（6 种）**：`zh-CN`（源语言）、`zh-TW`、`ja`、`en`、`de`、`fr`。
- **核心原则**：源语言 catalog（zh-CN）是唯一真源；其余语言均由 Crowdin 产出并回流入库。

### 范围（P1）

**包含：**

- i18n 框架接入与初始化（i18next + react-i18next + ICU + 语言探测）
- 共享 `AppLanguage` 类型与 locale 归一逻辑
- `uiStore` 新增 `locale` 状态（仿现有 `theme` 持久化范式）
- 语言切换器组件 `LanguageToggle`，挂载到两个页面 header
- 命名空间划分与目录结构
- i18next-parser 抽取配置
- Crowdin 全自动翻译流水线（含 GitHub Action）
- 后端语言传递接缝（前端注入站内 locale + Worker 中间件解析）
- Konva 画布 locale 感知 `fontFamily` + Canvas 内中文抽取
- **1 个样板文件**（HomePage 头部区域）的完整抽取，确立批量抽取范式

**不包含（明确排除）：**

- 380 条 UI 文案的全量抽取 —— P1 之后作为机械批量任务单独跟踪。
- 静态数据双语化（技能名/状态名/职业名/副本名）—— P2，语料从 XIVAPI（en/de/fr/ja 官方名）+ OpenCC（zh-TW）获取。
- 自托管 Web 字体的渲染一致性优化 —— P3，仅在系统字体栈不足时评估。

## 2. 全局约束

- 前端 React 19 + TS 5.9（`strict` + `verbatimModuleSyntax` + `erasableSyntaxOnly`）：类型导入必须 `import type`，禁用需 runtime transform 的语法（enum/namespace）与编译期宏。
- 包管理器：**pnpm**。
- 提交信息/作者/Co-Authored-By **禁止**包含 "Claude"（`.husky/commit-msg` 会拒绝）。
- 纯客户端 SPA（无 SSR），无需服务端语言协商。
- 后端 Worker 框架为 **Hono**（`src/workers/index.ts` + `src/workers/routes/*.ts`）。
- 测试：Vitest，测试文件与源文件同目录 `*.test.ts`；Worker 测试用 `vitest.workers.config.ts`。

## 3. 技术选型

| 依赖                               | 用途                                               | 类型          |
| ---------------------------------- | -------------------------------------------------- | ------------- |
| `i18next` + `react-i18next`        | 纯运行时 i18n 核心（`t()` 框架无关，可用于 Konva） | dependency    |
| `i18next-browser-languagedetector` | 首访 localStorage → `navigator.language` 回退      | dependency    |
| `i18next-icu`                      | en/de/fr 复数与 ICU MessageFormat 插值             | dependency    |
| `i18next-parser`                   | 从源码抽取 zh-CN catalog                           | devDependency |
| `@crowdin/cli`                     | 本地/CI 同步 Crowdin                               | devDependency |
| `crowdin/github-action`            | CI 自动上传源串 + 回流译文 PR                      | GitHub Action |

**选型理由**：react-i18next 纯运行时，完全规避严格 TS 工具链与编译期宏（如 Lingui macro）的冲突；其 `i18n.t()` 可在 Konva/store/工具函数等组件外随手调用，契合本项目 Canvas 渲染需求。

## 5. 状态与数据流

### 5.1 locale 状态（仿 theme 范式）

`uiStore` 已有 `theme` 的成熟持久化范式（模块级 `applyTheme`/`getInitialTheme` + 手写 localStorage key + `partialize` 排除）。locale 完全照搬：

```
首次加载:
  getInitialLocale()  // localStorage 'locale'(合法则用) → normalizeLocale(navigator.language) → DEFAULT_LOCALE('zh-CN')
  → 初始化 i18n.language + uiStore.locale

用户切换:
  LanguageToggle → uiStore.setLocale(l)
  → applyLocale(l)  // localStorage.setItem('locale', l) + i18n.changeLanguage(l)
  → React 组件经 react-i18next 重渲染；Konva 经 store 订阅重绘
```

- **语言不进 URL**：避开 OAuth `/callback` 固定回调约束，改动面最小，与 `theme` 同构。
- `locale` 从 `ui-store` 的 persist `partialize` 中**排除**（与 `theme` 一致，用独立 localStorage key `'locale'`，避免双重存储）。

### 5.2 normalizeLocale 归一规则

输入任意 `navigator.language` 或存储值，输出合法 `AppLanguage`：

| 输入示例                                     | 输出                      |
| -------------------------------------------- | ------------------------- |
| `zh-TW` / `zh-HK` / `zh-Hant` / `zh-Hant-TW` | `zh-TW`                   |
| `zh` / `zh-CN` / `zh-Hans` / 其他 `zh*`      | `zh-CN`                   |
| `en` / `en-US` / `en-GB`                     | `en`                      |
| `ja` / `ja-JP`                               | `ja`                      |
| `de` / `de-DE`                               | `de`                      |
| `fr` / `fr-FR`                               | `fr`                      |
| 未知 / `undefined` / `''`                    | `zh-CN`（DEFAULT_LOCALE） |

## 7. 翻译流水线（Crowdin 全自动）

```
开发写中文 JSX
  → i18next-parser 抽取 → src/i18n/locales/zh-CN/*.json（入库，唯一真源）
  → [GitHub Action] push 触发 crowdin upload sources
  → Crowdin 内：MT 预翻译（简→繁 / ja / en / de / fr）+ 人工校对
  → [GitHub Action] 自动开「译文回流 PR」（download zh-TW/ja/en/de/fr）
  → 人工 review PR 后合并（机翻不直接进主干）
```

### crowdin.yml 要点

- 源：`/src/i18n/locales/zh-CN/*.json`；`translation`: `/src/i18n/locales/%locale%/%original_file_name%`。
- **中文变体处理**：必须用 `%locale%` + `languages_mapping` 把 Crowdin 语言码映射到本项目目录名（`%two_letters_code%` 会把 zh-CN/zh-TW 折叠成 `zh` 而冲突）：

  ```yaml
  languages_mapping:
    locale:
      zh-TW: 'zh-TW'
      ja: 'ja'
      en: 'en'
      de: 'de'
      fr: 'fr'
  ```

- token 走 `api_token_env: CROWDIN_PERSONAL_TOKEN`，**绝不入库**。

### CI 与 Secrets

- GitHub Action（`crowdin/github-action`）：push 自动 `upload_sources: true`，并 `download_translations: true` + `create_pull_request: true` 回流。
- Secrets：`CROWDIN_PROJECT_ID`、`CROWDIN_PERSONAL_TOKEN`（GitHub Secrets）。

### 前置（人工，spec 外的运维步骤）

1. 在 Crowdin 创建项目，源语言设为 Chinese Simplified。
2. 勾选 5 个目标语言：zh-TW、ja、en、de、fr。
3. （可选）配置 MT 引擎用于预翻译。
4. 在 GitHub 仓库配置上述两个 Secrets。

## 8. 后端语言传递接缝（预留，不改变现有输出）

**目的**：把"客户端站内 locale → API 请求 → Worker → FFLogs 取数逻辑"的管道在 P1 一次铺好，未来 FFLogs 导入按语言变更逻辑时无需重新铺线。

### 前端注入

`src/api/apiClient.ts` 的 `beforeRequest` hook 统一注入标准头 `Accept-Language: <uiStore 当前 locale>`（标准头，CORS-safelisted，无需改 CORS allowlist）。

**为什么需要前端注入**：浏览器虽自动带 `Accept-Language`，但那是 OS/浏览器语言，与 app 内 `LanguageToggle` 选定的 locale 会分叉（如中文系统用户站内切到 English）。前端注入用站内 locale **覆盖**浏览器默认头，保证后端读到用户真实选择。

### 后端读取

- 新增 Hono 中间件 `src/workers/middleware/readLanguage.ts`：读 `c.req.header('Accept-Language')`，经共享 `normalizeLocale` 解析为 `AppLanguage`，写入 `c`（`c.set('lang', ...)`），与现有 `tryReadAuth` 中间件同构。挂载到 `/api/fflogs` 子路由。
- 各 fflogs handler 从上下文取用（`c.get('lang')`），向下传给 `fflogsClientV2` 的 `getReport`/`getEvents`。

### 本期边界：只铺线不分支

- `getEvents` 现有的死参数 `lang`（`fflogsClientV2.ts:528` 未解构）接上，使其从 handler 一路可达 client 内部。
- 两处硬编码 `translate: false`（`fflogsClientV2.ts:380`、`:539`）**保持不变**，FFLogs 输出与现状完全一致。
- 在 client 内 `translate` 决策处留一处带注释的扩展点，标注未来可按 `lang` 切换 FFLogs 取数行为。
- `src/types/fflogs.ts` 与 `fflogsClientV2.ts` 的 `lang?: string` 字段统一收敛到共享 `AppLanguage` 类型。

> **勿按「死参数」清理**：main 的 `7c9894c` 曾以「GraphQL 层硬编码 `translate: false`，lang 从未生效」
> 为由摘除整条 lang 穿透链（前端 `fflogsClient` → 路由 → `fflogsClientV2`）。该链在本 spec 中是
> **P2 数据双语化的刻意预留**，rebase 到该 commit 之后已重新接回。若后续再遇到「lang 无消费点」的
> 清理提案，应先确认 P2 是否已放弃，而非直接删除。

## 10. 字体策略（系统字体栈）

- DOM：Tailwind 字体栈含中（简/繁）/日的系统字体 fallback 链。
- Konva：`getKonvaFontFamily(locale)` 按 locale 返回含对应 CJK 系统字体的字体栈。
- 不打包自托管 CJK 字体（数百 KB~MB 级，冲突于现有 PWA/分包优化目标）；渲染一致性问题留 P3 评估。
