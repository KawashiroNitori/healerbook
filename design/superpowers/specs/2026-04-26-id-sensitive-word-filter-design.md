# Timeline ID 敏感词过滤设计

**日期**：2026-04-26
**状态**：设计已批准

## 背景

`POST /api/timelines` 在 Worker 中用 `generateId()`（nanoid 21 位 base62）生成共享时间轴 ID。该 ID 直接出现在地址栏（`/timeline/:id`）。需要保证生成的 ID 不包含任何敏感词子串（不区分大小写）。

额外约束：

1. 敏感词列表**不能以明文出现在代码库中**
2. 运行时**不能通过 console 打印敏感词明文**

## 范围

**仅 Worker 侧生效**：只对 `src/workers/timelines.ts:handlePost` 生成的共享 timeline ID 过滤。
**不在前端生效**：本地新建（`createNewTimeline`）、FFLogs 导入、Editor 派生 ID 等前端路径不过滤。前端 bundle 内不含任何敏感词痕迹（明文与哈希皆不含）。

理由：URL 暴露面真正起公共风险的是已发布的共享 timeline。本地 ID 仅本人浏览器可见，过滤价值低、复杂度高（需把哈希列表打到客户端）。

## 关键决策

| 维度          | 选择                                                                                                                          |
| ------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 作用域        | Worker only                                                                                                                   |
| 词表存储      | **本机**用 HMAC-SHA256 预哈希、按词长分组，输出**已 commit 入 git**的生成模块；明文 txt 留在本机不入 git                      |
| 哈希生成时机  | 显式手动命令 `pnpm gen:sensitive-words`（不挂任何 prebuild/predev hook，避免本地无 txt 时静默覆盖已 commit 的产物）           |
| CI 角色       | 不参与，不需要词表也不需要构建期 key                                                                                          |
| HMAC 密钥     | `SENSITIVE_WORDS_HMAC_KEY`，两处同值（本地 `.dev.vars` 用于 wrangler dev + 本机生成；Workers Secret 用于线上 worker runtime） |
| 匹配规则      | 子串匹配，不区分大小写                                                                                                        |
| 命中处理      | Worker 内重试 32 次，全失败返回 500                                                                                           |
| 词表/key 缺失 | 生成模块若为空表则运行时 no-op，**首次调用 log 一次健康状态**                                                                 |
| 漂移检测      | 生成模块带 `KEY_FINGERPRINT`；runtime 首次调用核对，对不上 log 一次 warn，不影响 fail-open 行为                               |
| 历史 ID       | 不扫不改，仅对未来 POST 生效                                                                                                  |

## 威胁模型

**保护目标**：词表明文不外泄。

| 攻击面                   | 措施                                                                                                                 |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------- |
| 翻代码库 / git history   | 词表 txt 在 `secrets/`，加入 `.gitignore`，仅本机存在；commit 入 git 的只有 HMAC 哈希集合                            |
| 翻部署后的 Worker bundle | bundle 中含哈希集合（与 git 中的相同），但 HMAC key 不在 bundle（独立的 Workers Secret），无法离线反推（即使是短词） |
| 运行日志                 | 过滤器内不 log 词、不 log 哈希、不 log 命中位置；命中只回退到重试计数                                                |
| HMAC key 自身            | 两处同值，每处独立 secret 渠道（CF Workers Secret / 本地 `.dev.vars`），均不入 git                                   |

**不在保护范围**：HMAC key 本身泄露，则攻击者可枚举哈希反推词表（这是不可避免的对偶）。视为业务可接受风险。

## 模块设计

### 1. `scripts/build-sensitive-words.ts`

**手动运行**的 Node 脚本，绑定到 `package.json` 的 `"gen:sensitive-words"` script。**不**挂 prebuild/predev hook（避免本地无 txt 时静默生成空表覆盖已 commit 的产物）。

**输入**：

- `process.env.SENSITIVE_WORDS_HMAC_KEY`：HMAC 密钥（脚本同时读取项目根 `.dev.vars` 作为 fallback；简单 `KEY=VALUE` 行解析，忽略注释与空行）
- `secrets/sensitive-words.txt`：每行一个敏感词，支持空行 / `#` 起始的注释行

**输出**：`src/workers/sensitiveWordHashes.generated.ts`

```ts
// AUTO-GENERATED by scripts/build-sensitive-words.ts. DO NOT EDIT.
export const HASHES: ReadonlyMap<number, ReadonlySet<string>> = new Map([
  [3, new Set(['Bx7s+w==' /* ... */])],
  [
    4,
    new Set([
      /* ... */
    ]),
  ],
  // ...
])
export const LENGTHS: readonly number[] = [3, 4, 5 /* ... */]
// HMAC(key, "sensitive-words-fingerprint-v1") 截断 16 字节 base64
// 让 runtime 校验：当前 worker 持有的 key 是否就是生成此哈希表用的同一把
export const KEY_FINGERPRINT: string = '...'
```

**逻辑**：

1. 读 key（env 优先，`.dev.vars` fallback）；**缺则报错退出 1**（手动命令，必须显式提供，否则会无声破坏哈希一致性）
2. 读 txt；缺则**输出空表**并 log `Sensitive word file not found, generated empty table`，退出 0（用于首次脚手架阶段）
3. 逐行 `trim().toLowerCase()`，跳过空与 `#` 起始行，去重
4. `crypto.createHmac('sha256', key)` 计算每词的哈希，base64 截断到 16 字节（128 bit，碰撞可忽略）
5. 按词长分组
6. 计算 `KEY_FINGERPRINT = HMAC(key, "sensitive-words-fingerprint-v1")[:16].base64()`，固定字符串作为 magic input，仅依赖 key 不依赖词表（不暴露 key）
7. 写入生成模块
8. **不 log 任何词、哈希、key；只输出 `Generated N hashes across M length buckets` 或前述空表提示**

**幂等性**：每次运行覆写生成模块；输出按 length 升序、每 length 内 hash 字典序，保证相同输入产物 byte-equal（git diff 干净）。

**与 git 的关系**：开发者改完 txt 后**必须**手动运行此脚本并把生成模块一起 commit。pre-commit hook 不强制（避免误触发，开发者自负）；但 review / lint 可针对 `*.generated.ts` 设规则提示。

### 2. `src/workers/sensitiveWordFilter.ts`

```ts
import { HASHES, LENGTHS, KEY_FINGERPRINT } from './sensitiveWordHashes.generated'

export interface SensitiveWordEnv {
  SENSITIVE_WORDS_HMAC_KEY?: string
}

const FINGERPRINT_MAGIC = 'sensitive-words-fingerprint-v1'

let cachedHmacKey: CryptoKey | null = null
let cachedKeyMaterial: string | null = null
let healthChecked = false // 健康状态 + drift 校验只跑一次

async function getHmacKey(material: string): Promise<CryptoKey> {
  if (cachedHmacKey && cachedKeyMaterial === material) return cachedHmacKey
  cachedKeyMaterial = material
  cachedHmacKey = await crypto.subtle.importKey(
    'raw',
    new TextEncoder().encode(material),
    { name: 'HMAC', hash: 'SHA-256' },
    false,
    ['sign']
  )
  return cachedHmacKey
}

function hashToBase64Trunc(buf: ArrayBuffer): string {
  const bytes = new Uint8Array(buf, 0, 16)
  let bin = ''
  for (let i = 0; i < bytes.length; i++) bin += String.fromCharCode(bytes[i])
  return btoa(bin)
}

async function runHealthCheck(env: SensitiveWordEnv): Promise<void> {
  if (healthChecked) return
  healthChecked = true

  if (!env.SENSITIVE_WORDS_HMAC_KEY) {
    console.warn('[sensitive-words] disabled: missing SENSITIVE_WORDS_HMAC_KEY')
    return
  }
  if (LENGTHS.length === 0) {
    console.warn('[sensitive-words] disabled: empty hash table')
    return
  }

  // drift 校验：用当前 runtime key 算 fingerprint，对比生成期写死的值
  const key = await getHmacKey(env.SENSITIVE_WORDS_HMAC_KEY)
  const sig = await crypto.subtle.sign('HMAC', key, new TextEncoder().encode(FINGERPRINT_MAGIC))
  const fp = hashToBase64Trunc(sig)
  if (fp !== KEY_FINGERPRINT) {
    console.warn(
      '[sensitive-words] key drift: runtime key does not match generated table; filter will not match anything'
    )
    return
  }

  let total = 0
  for (const L of LENGTHS) total += HASHES.get(L)?.size ?? 0
  console.log(`[sensitive-words] active: ${total} entries across ${LENGTHS.length} lengths`)
}

export async function containsBannedSubstring(id: string, env: SensitiveWordEnv): Promise<boolean> {
  await runHealthCheck(env)

  if (!env.SENSITIVE_WORDS_HMAC_KEY || LENGTHS.length === 0) return false

  const key = await getHmacKey(env.SENSITIVE_WORDS_HMAC_KEY)
  const lower = id.toLowerCase()
  const enc = new TextEncoder()

  for (const L of LENGTHS) {
    const bucket = HASHES.get(L)
    if (!bucket) continue
    const limit = lower.length - L
    for (let i = 0; i <= limit; i++) {
      const sub = lower.slice(i, i + L)
      const sig = await crypto.subtle.sign('HMAC', key, enc.encode(sub))
      if (bucket.has(hashToBase64Trunc(sig))) return true
    }
  }
  return false
}
```

**性能**：21 位 ID × 假设 20 种词长 ≈ 420 次 HMAC-SHA256，单次约百微秒，总耗时 ms 量级。`runHealthCheck` 只跑一次（module 生命周期内）。

**注意**：

- `cachedHmacKey` 为 module-scope，跨请求复用；material 变化重新派生
- `healthChecked` 也是 module-scope，每个 worker isolate 启动后只跑一次健康检查与 drift 校验
- drift 校验只 log 不 throw、不改变 fail-open 行为
- 不 log 词、哈希、key、命中位置；只 log 表大小（来源于已 commit 的 generated 模块，本身就是公开的）
- 显式 `enc.encode(sub)`：sub 全部为 base62 ASCII，UTF-8 单字节安全
- drift 时仍走完整滑窗匹配（用错的 key 不会命中任何东西，逻辑等价 no-op，性能略多花一些 HMAC——可接受，因为这是不该发生的状态）

### 3. `src/workers/timelines.ts:handlePost` 改动

```ts
import { containsBannedSubstring } from './sensitiveWordFilter'

const ID_GEN_MAX_ATTEMPTS = 32

async function generateCleanId(env: Env): Promise<string> {
  for (let i = 0; i < ID_GEN_MAX_ATTEMPTS; i++) {
    const id = generateId()
    if (!(await containsBannedSubstring(id, env))) return id
  }
  throw new IdGenerationError()
}

class IdGenerationError extends Error {}

// handlePost 内：
try {
  const newId = await generateCleanId(env)
  // ... 原有 INSERT 与返回 ...
} catch (e) {
  if (e instanceof IdGenerationError) {
    return jsonRes({ error: 'id_generation_failed' }, 500, env.ALLOWED_ORIGIN)
  }
  throw e
}
```

`Env` interface 中加 `SENSITIVE_WORDS_HMAC_KEY?: string` 可选字段。

## 错误处理矩阵

| 场景                                                          | 行为                                                                                                   |
| ------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| `secrets/sensitive-words.txt` 缺（首次脚手架/clean checkout） | 生成脚本输出空哈希表；运行时首次调用 log `[sensitive-words] disabled: empty hash table`                |
| `SENSITIVE_WORDS_HMAC_KEY` 生成期缺                           | **报错退出 1**（手动命令必须显式提供 key）                                                             |
| `SENSITIVE_WORDS_HMAC_KEY` 运行期缺                           | 过滤器直接 `return false`；首次调用 log `[sensitive-words] disabled: missing SENSITIVE_WORDS_HMAC_KEY` |
| 哈希表与 runtime key 不匹配（drift）                          | 首次调用 log `[sensitive-words] key drift: ...` warn 一次；fail-open（不抛错），过滤事实上失效         |
| 已 commit 的哈希模块被误删                                    | TS 编译报错；开发者重跑 `pnpm gen:sensitive-words`                                                     |
| 32 次重试均命中                                               | `handlePost` 返 500 `{ error: 'id_generation_failed' }`；不 log 详情                                   |
| HMAC 计算抛异常                                               | 冒泡到 `handlePost`，返 500 `{ error: 'Internal' }`                                                    |
| txt 含空行 / `#` 注释                                         | 构建脚本跳过                                                                                           |
| txt 含全角字符                                                | 不特殊处理。base62 ID 不会包含非 ASCII，全角词永远不命中（即使写进去也无害）                           |

## 测试

### `sensitiveWordFilter.test.ts`

不依赖真实 secrets。通过 vitest module mock 替换 `./sensitiveWordHashes.generated`：

- 命中精确子串：注入哈希一个测试词，验证含该词的 ID 命中
- 命中位置：起始 / 中间 / 末尾各一个 case
- 大小写无关：词写小写，ID 大写，仍命中
- 不命中：随机 ID 不命中
- 空 LENGTHS / 空 HASHES：no-op 返 false，首次调用 log `disabled: empty hash table`
- 缺 `SENSITIVE_WORDS_HMAC_KEY`：no-op 返 false（即便表非空），首次调用 log `disabled: missing ...`
- key drift：mock `KEY_FINGERPRINT` 写一个值，传入不同 key，首次调用 log `key drift: ...` 且后续匹配返 false
- 健康日志只跑一次：连调两次，console.log/warn 只触发一次

测试用的"敏感词"夹具是无害字串（如 `"banword"`），不 commit 任何真实词。

### `timelines.test.ts` 增量

- mock `containsBannedSubstring` 让前 N 次返 true、第 N+1 次返 false：验证 `handlePost` 真的重试且最终返回干净 ID
- mock 让 32 次都返 true：验证返回 500 `{ error: 'id_generation_failed' }`
- mock 让 0 次命中：验证现有路径不受影响（兼容性）

### `build-sensitive-words.ts` 测试（可选）

由于纯构建脚本、行为简单（读文件 → HMAC → 写文件），优先级低。可手测：

- `secrets/sensitive-words.txt` 缺 + key 在 → 生成空表（exit 0），KEY_FINGERPRINT 仍写入
- key 缺 → 报错 exit 1
- 二者都在 → 生成非空表，二次运行产物 byte-equal
- 同 txt 不同 key → 生成的 KEY_FINGERPRINT 不同，所有 hash 也不同

### Benchmark

新增 `src/workers/sensitiveWordFilter.bench.ts`，用 vitest bench API（`bench(...)`）。**仅本地手动**跑（`pnpm vitest bench`），不进 CI。

**直接用真实生成数据**：bench 直接 `import { HASHES, LENGTHS } from './sensitiveWordHashes.generated'` 跑，因为：

- 本地已有真实哈希表（commit 入 git）
- 真实数据反映实际线上规模与词长分布，比合成伪词更准
- 想看不同规模的影响时，开发者临时改 `secrets/sensitive-words.txt` 重跑 `pnpm gen:sensitive-words` 即可

**两个 bench**：

1. `containsBannedSubstring(randomId, env)` —— 单次过滤开销
2. `generateCleanId(env)` —— 完整 ID 生成（含 nanoid + 过滤）

每个 bench 内随机生成 21 位 base62 ID 喂进去（`generateId()`），跑迭代统计。

**预算**（基于经验，超出说明实现退化）：

- `containsBannedSubstring` < 5ms / 次
- `generateCleanId` 同量级（命中概率极低，几乎等于 1× 过滤 + 1× nanoid）
- 任一指标 > 50ms 视为退化，需重新审实现

## 密钥与词表的注入路径

| 阶段                                       | 需要                                                       | 来源                                        |
| ------------------------------------------ | ---------------------------------------------------------- | ------------------------------------------- |
| 本机生成哈希（`pnpm gen:sensitive-words`） | `SENSITIVE_WORDS_HMAC_KEY` + `secrets/sensitive-words.txt` | env / `.dev.vars` + 本地文件                |
| `pnpm dev`（wrangler dev）                 | `env.SENSITIVE_WORDS_HMAC_KEY`                             | `.dev.vars` 注入                            |
| `pnpm build` / `wrangler deploy`           | 无（哈希表已 commit；运行期 key 由 Workers Secret 提供）   | —                                           |
| Worker 运行期                              | `env.SENSITIVE_WORDS_HMAC_KEY`                             | `wrangler secret put` 配置的 Workers Secret |

CI 完全不参与本特性：它拿到的 repo 已经包含哈希模块，正常 build & deploy 即可。

## 部署 Runbook

**初次部署（一次性）：**

1. 生成 32 字节随机串（`openssl rand -base64 32`）
2. `pnpm dlx wrangler secret put SENSITIVE_WORDS_HMAC_KEY` → 粘贴该值
3. 本地 `.dev.vars` 添加 `SENSITIVE_WORDS_HMAC_KEY=...`，值同上
4. 本地 `secrets/sensitive-words.txt` 放词表（每行一个词）
5. `pnpm gen:sensitive-words` 生成 `src/workers/sensitiveWordHashes.generated.ts`
6. `git add src/workers/sensitiveWordHashes.generated.ts` + commit + push
7. CI 走正常 build/deploy 流程，无需额外配置

**词表更新**：改本地 txt → `pnpm gen:sensitive-words` → commit 生成模块 → push → CI 部署。

## 不做的事（YAGNI）

- 不做 leetspeak / 字符替换变体匹配
- 不做正则 / 通配符词条
- 不做客户端过滤
- 不扫描历史 D1 数据
- 不提供「绕过过滤」的运维开关（缺 key/缺词表本身就是 no-op）
